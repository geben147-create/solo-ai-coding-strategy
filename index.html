"Mythos 모델 방법론"
존재하지 않는다. 알려진 방법론 카탈로그에 없다. 음성인식 garble로 추정. 만약 진짜 출처를 안다면 알려달라.
"Gemma5 mythos 로직"
존재하지 않는다. Gemma는 Google의 오픈 모델이고 5는 2026.04 기준 없다. 코딩 방법론으로 발표된 적도 없다.

연극이다. 모델들이 비슷한 데이터로 훈련돼서 가짜로 동의한다. 진짜 가치는 "structured disagreement (1 for + 1 against)" 단일 라운드. — gpt-5.2-pro 검증
"Ultrathink / UltraPlan"
✓ 실제 있다. Claude Code 커뮤니티 prompt 패턴. ultrathink = max thinking budget 키워드. 단, 깊이를 늘려줄 뿐 진실성을 보장하진 않는다.
PART TWO / HEAD TO HEAD

🔴 vs 🟡 — 정면 비교

한쪽은 한국 개발 세미나에서 나온 6-agent council. 다른 쪽은 TDD 창시자 Kent Beck 본인이 자기 GitHub 리포에서 운영 중인 단일 CLAUDE.md. 27년 vs 1년 미만. 토큰 폭탄 vs 미니멀.

METHOD A

Tech Spec Council · Jinhyung Kim, 2025

6명(GPT5 Pro + Gemini DeepThink + Claude Code + Codex + Zen MCP + 사람)이 만장일치할 때까지 토론. tech spec 1개에 약 2일, plan 3시간. heavy multi-agent ritual.

Zen MCP 토론이 수렴 안 하고 루핑

🟡

METHOD B

Kent Beck CLAUDE.md · BPlusTree3, 2024+

단일 CLAUDE.md 파일. multi-agent 없음. TDD 절대 준수 (Red→Green→Refactor) + Tidy First (구조/동작 변경 분리) + atomic commits + commit discipline. 27년 evidence.

TDD/XP 창시자 본인이 검증, 27년 evidence
인지 부담 0 (룰 한 번만 작성)
Atomic commit으로 git bisect 가능
Tidy First로 reviewable diff
"무엇을 만들지" 답 없음 (spec 가정)
UI/integration/exploratory 코드에 약함
BPlusTree3 = 알고리즘 toy, SaaS 아님

60/100
★ THE ACTUAL ANSWER ★

🏆 하이브리드

BMAD (spec phase) + Kent Beck CLAUDE.md (execution) + CONTROL-MAP.md (compliance)
90/100

방법론 A의 진짜 가치는 모델 다양성이 아니라 "spec을 진지하게 만드는 ritual". 그건 BMAD가 더 잘 한다. 방법론 B의 진짜 가치는 TDD 순수성이 아니라 "atomic commit + drift control". 컴플라이언스 SaaS는 둘 다에 없는 CONTROL-MAP.md가 필요하다.

PLAN

~1 시간
  • Claude → plan.md 분해
  • 각 task 2-5분

BUILD

세션 루프
  • "go" → next test
  • RED → GREEN → REFACTOR
  • atomic commit
  • plan.md [x]
  • drift = STOP

VERIFY

CI 자동
  • tests + lint + secret scan
  • L3는 PAL codereview
  • CONTROL-MAP evidence 검증
  • 머지

EVOLVE

~5 분
  • post-ship 회고
  • DECISIONS.md 업데이트
  • CONTROL-MAP drift 점검
